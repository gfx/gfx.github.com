329=head1 NAME
330
331Plack::Request - Portable HTTP request object from PSGI env hash
332
333=head1 SYNOPSIS
334
335 use Plack::Request;
336
337 my $app_or_middleware = sub {
338 my $env = shift; # PSGI env
339
340 my $req = Plack::Request->new($env);
341
342 my $path_info = $req->path_info;
343 my $query = $req->param('query');
344
345 my $res = $req->new_response(200); # new Plack::Response
346 $res->finalize;
347 };
348
349=head1 DESCRIPTION
350
351L<Plack::Request> provides a consistent API for request objects across
352web server environments.
353
354=head1 CAVEAT
355
356Note that this module is intended to be used by Plack middleware
357developers and web application framework developers rather than
358application developers (end users).
359
360Writing your web application directly using Plack::Request is
361certainly possible but not recommended: it's like doing so with
362mod_perl's Apache::Request: yet too low level.
363
364If you're writing a web application, not a framework, then you're
365encouraged to use one of the web application frameworks that support PSGI (L<http://plackperl.org/#frameworks>),
366or see modules like L<HTTP::Engine> to provide higher level
367Request and Response API on top of PSGI.
368
369=head1 METHODS
370
371Some of the methods defined in the earlier versions are deprecated in
372version 0.99. Take a look at L</"INCOMPATIBILITIES">.
373
374Unless otherwise noted, all methods and attributes are B<read-only>,
375and passing values to the method like an accessor doesn't work like
376you expect it to.
377

575=head2 Hash::MultiValue parameters
576
577Parameters that can take one or multiple values (i.e. C<parameters>,
578C<query_parameters>, C<body_parameters> and C<uploads>) store the
579hash reference as a L<Hash::MultiValue> object. This means you can use
580the hash reference as a plain hash where values are B<always> scalars
581(B<NOT> array references), so you don't need to code ugly and unsafe
582C<< ref ... eq 'ARRAY' >> anymore.
583
584And if you explicitly want to get multiple values of the same key, you
585can call the C<get_all> method on it, such as:
586
587 my @foo = $req->query_parameters->get_all('foo');
588
589You can also call C<get_one> to always get one parameter independent
590of the context (unlike C<param>), and even call C<mixed> (with
591Hash::MultiValue 0.05 or later) to get the I<traditional> hash
592reference,
593
594 my $params = $req->parameters->mixed;
595
596where values are either a scalar or an array reference depending on
597input, so it might be useful if you already have the code to deal with
598that ugliness.
599
600=head2 PARSING POST BODY and MULTIPLE OBJECTS
601
602The methods to parse request body (C<content>, C<body_parameters> and
603C<uploads>) are carefully coded to save the parsed body in the
604environment hash as well as in the temporary buffer, so you can call
605them multiple times and create Plack::Request objects multiple times
606in a request and they should work safely, and won't parse request body
607more than twice for the efficiency.
608
609=head1 DISPATCHING
610
611If your application or framework wants to dispatch (or route) actions
612based on request paths, be sure to use C<< $req->path_info >> not C<<
613$req->uri->path >>.
614
615This is because C<path_info> gives you the virtual path of the request,
616regardless of how your application is mounted. If your application is
617hosted with mod_perl or CGI scripts, or even multiplexed with tools
618like L<Plack::App::URLMap>, request's C<path_info> always gives you
619the action path.
620
621Note that C<path_info> might give you an empty string, in which case
622you should assume that the path is C</>.
623
624You will also want to use C<< $req->base >> as a base prefix when
625building URLs in your templates or in redirections. It's a good idea
626for you to subclass Plack::Request and define methods such as:
627
628 sub uri_for {
629 my($self, $path, $args) = @_;
630 my $uri = $self->base;
631 $uri->path($uri->path . $path);
632 $uri->query_form(@$args) if $args;
633 $uri;
634 }
635
636So you can say:
637
638 my $link = $req->uri_for('/logout', [ signoff => 1 ]);
639
640and if C<< $req->base >> is C</app> you'll get the full URI for
641C</app/logout?signoff=1>.
642
643=head1 INCOMPATIBILITIES
644
645In version 0.99, many utility methods are removed or deprecated, and
646most methods are made read-only.
647
648The following methods are deprecated: C<hostname>, C<url_scheme>,
649C<params>, C<query_params>, C<body_params>, C<cookie> and
650C<raw_uri>. They will be removed in the next major release.
651
652All parameter-related methods such as C<parameters>,
653C<body_parameters>, C<query_parameters> and C<uploads> now contains
654L<Hash::MultiValue> objects, rather than I<scalar or an array
655reference depending on the user input> which is insecure. See
656L<Hash::MultiValue> for more about this change.
657
658C<< $req->path >> method had a bug, where the code and the document
659was mismatching. The document was suggesting it returns the sub
660request path after C<< $req->base >> but the code was always returning
661the absolute URI path. The code is now updated to be an alias of C<<
662$req->path_info >> but returns C</> in case it's empty. If you need
663the older behavior, just call C<< $req->uri->path >> instead.
664
665Cookie handling is simplified, and doesn't use L<CGI::Simple::Cookie>
666anymore, which means you B<CAN NOT> set array reference or hash
667reference as a cookie value and expect it be serialized. You're always
668required to set string value, and encoding or decoding them is totally
669up to your application or framework. Also, C<cookies> hash reference
670now returns I<strings> for the cookies rather than CGI::Simple::Cookie
671objects, which means you no longer have to write a wacky code such as:
672
673 $v = $req->cookie->{foo} ? $req->cookie->{foo}->value : undef;
674
675and instead, simply do:
676
677 $v = $req->cookie->{foo};
