61=head1 DESCRIPTION
62
63Plack::Middleware is a utility base class to write PSGI
64middleware. All you have to do is to inherit from Plack::Middleware
65and then implement the callback C<call> method (or C<to_app> method
66that would return the PSGI code reference) to do the actual work. You
67can use C<< $self->app >> to call the original (wrapped) application.
68
69Your middleware object is created at a PSGI application compile time
70and is persistent during the web server life cycle (unless it is a
71non-persistent environment such as CGI), so you should never set or
72cache per-request data like C<$env> in your middleware object. See
73also L<Plack::Component/"OBJECT LIFECYCLE">.
74
75See L<Plack::Builder> how to actually enable middleware in your
76I<.psgi> application file using the DSL. If you do not like our
77builder DSL, you can also use C<wrap> method to wrap your application
78with a middleware:
79
80 use Plack::Middleware::Foo;
81
82 my $app = sub { ... };
83 $app = Plack::Middleware::Foo->wrap($app, %options);
84 $app = Plack::Middleware::Bar->wrap($app, %options);
85
86=head1 RESPONSE CALLBACK
87
88The typical middleware is written like this:
89
90 package Plack::Middleware::Something;
91 use parent qw(Plack::Middleware);
92
93 sub call {
94 my($self, $env) = @_;
95 # pre-processing $env
96 my $res = $self->app->($env);
97 # post-processing $res
98 return $res;
99 }
100
101The tricky thing about post processing the response is that it could
102either be an immediate 3 element array ref, or a code reference that
103implements the delayed (streaming) interface.
104
105Dealing with these two types of response in each piece of middleware
106is pointless, so you're recommended to use the C<response_cb> wrapper
107function in L<Plack::Util> when implementing a post processing
108middleware.
109
110 my $res = $app->($env);
111 Plack::Util::response_cb($res, sub {
112 my $res = shift;
113 # do something with $res;
114 });
115
116The callback function gets a PSGI response as a 3 element array
117reference, and you can update the reference to implement the post
118processing.
119
120 package Plack::Middleware::Always500;
121 use parent qw(Plack::Middleware);
122 use Plack::Util;
123
124 sub call {
125 my($self, $env) = @_;
126 my $res = $self->app->($env);
127 Plack::Util::response_cb($res, sub {
128 my $res = shift;
129 $res->[0] = 500;
130 return;
131 });
132 }
133
134In this example, the callback gets the C<$res> and updates its first
135element (status code) to 500. Using C<response_cb> makes sure that
136this works with the delayed response too.
137
138You're not required (and not recommended either) to return a new array
139reference - they will be simply ignored. You're suggested to
140explicitly return, unless you fiddle with the content filter callback
141(see below).
142
143Similarly, note that you have to keep the C<$res> reference when you
144swap the entire response.
145
146 Plack::Util::response_cb($res, sub {
147 my $res = shift;
148 $res = [ $new_status, $new_headers, $new_body ]; # THIS DOES NOT WORK
149 return;
150 });
151
152This does not work, since assigning a new anonymous array to C<$res>
153doesn't update the original PSGI response value. You should instead
154do:
155
156 Plack::Util::response_cb($res, sub {
157 my $res = shift;
158 @$res = ($new_status, $new_headers, $new_body); # THIS WORKS
159 return;
160 });
161
162The third element of PSGI response array ref is a body, and it could
163be either array ref or IO::Handle-ish object. The application could
164also make use of C<$writer> object if C<psgi.streaming> is in
165effect. Dealing with these variants is again really painful, and
166C<response_cb> can take care of that too, by allowing you to return a
167content filter as a code reference.
168
169 # replace all "Foo" in content body with "Bar"
170 Plack::Util::response_cb($res, sub {
171 my $res = shift;
172 return sub {
173 my $chunk = shift;
174 return unless defined $chunk;
175 $chunk =~ s/Foo/Bar/g;
176 return $chunk;
177 }
178 });
179
180The callback takes one argument C<$chunk> and your callback is
181expected to return the updated chunk. If the given C<$chunk> is undef,
182it means the stream has reached the end, so your callback should also
183return undef, or return the final chunk and return undef when called
184next time.
